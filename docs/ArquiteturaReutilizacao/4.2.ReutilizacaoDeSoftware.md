## 3. Teoria: Frameworks

### 3.1. O que Ă© um Framework?

Um framework Ă© um conjunto de classes e interfaces que define uma arquitetura reutilizĂĄvel para uma famĂ­lia de problemas de um mesmo domĂ­nio. Diferentemente de uma biblioteca de classes, em que **o cĂłdigo cliente chama a biblioteca**, em um framework o sentido se inverte: **o framework chama o cĂłdigo do cliente**, princĂ­pio conhecido como *Hollywood Principle* ("Don't call us, we'll call you").

Essa inversĂŁo de controle Ă© o que distingue fundamentalmente um framework de qualquer outro mecanismo de reutilizaĂ§ĂŁo:

| CaracterĂ­stica | Biblioteca de Classes | Framework |
| :------------- | :-------------------- | :-------- |
| **Controle do fluxo** | O cliente controla | O framework controla |
| **Conhecimento de domĂ­nio** | NĂŁo possui | Embutido no nĂșcleo |
| **InteraĂ§ĂŁo entre objetos** | Definida pelo cliente | Definida pelo framework |
| **Comportamento padrĂŁo** | NĂŁo fornece | Fornece (frozen spots) |
| **CustomizaĂ§ĂŁo** | O cliente instancia classes | O cliente estende hot spots |

### 3.2. Hot Spots e Frozen Spots

Todo framework Ă© estruturado em torno de dois tipos de partes:

**Frozen Spots** (Parte Fixa): definem a arquitetura geral do framework com seus componentes bĂĄsicos, os relacionamentos entre eles e o fluxo de controle. Permanecem idĂȘnticos em todas as instanciaĂ§Ă”es do framework. TambĂ©m chamados de "core" ou "kernel" do framework.

**Hot Spots** (Parte FlexĂ­vel): representam os pontos de variaĂ§ĂŁo do framework, projetados para serem adaptados Ă s necessidades de cada aplicaĂ§ĂŁo cliente. SĂŁo normalmente representados por classes abstratas, mĂ©todos abstratos ou interfaces que o cliente deve implementar.

### 3.3. Tipos de Framework

| Tipo | Mecanismo de ReutilizaĂ§ĂŁo | CaracterĂ­stica |
| :--- | :------------------------ | :------------- |
| **Caixa Branca** | HeranĂ§a e implementaĂ§ĂŁo de mĂ©todos abstratos | Orientado a hot spots; requer conhecimento interno |
| **Caixa Preta** | ComposiĂ§ĂŁo e consumo de serviĂ§os prontos | Orientado a frozen spots; mais simples de usar |
| **Caixa Cinza** | HeranĂ§a + composiĂ§ĂŁo (hĂ­brido) | Combina hot spots e frozen spots |

### 3.4. PadrĂ”es GoF Embutidos no Framework

Frameworks frequentemente incorporam padrĂ”es de projeto GoF em seu nĂșcleo. O `organizeseugrupo-framework` embute dois padrĂ”es jĂĄ documentados na Entrega 3:

#### Classe abstrata ProcessadorNotificacaoFramework (Template Method)

O domĂ­nio de notificaĂ§Ă”es foi escolhido porque faz sentido diretamente no OrganizeSeuGrupo: a aplicaĂ§ĂŁo coordena grupos, reuniĂ”es, materiais e participantes. ```typescript

```typescript
type CanalNotificacao = 'email' | 'app' | 'email-e-app';

    id: number;
    nome: string;
    email: string;
}

interface EventoNotificacao {
    grupo: string;
    titulo: string;
    autor: Usuario;
    participantes: Usuario[];
    responsaveis?: Usuario[];
    dados?: Record<string, unknown>;
}

interface ConteudoNotificacao {
    assunto: string;
    corpo: string;
    canal: CanalNotificacao;
}

interface RegistroNotificacao {
    destinatarios: Usuario[];
    conteudo: ConteudoNotificacao;
    prioridade: PrioridadeNotificacao;
    enviadaEm: Date;
}

abstract class ProcessadorNotificacaoFramework {
    public enviarNotificacao(evento: EventoNotificacao): RegistroNotificacao {

        this.validarDestinatarios(destinatarios);

        const conteudo = this.montarConteudo(evento);
        const prioridade = this.definirPrioridade(evento);
        const registro = this.despachar(destinatarios, conteudo, prioridade);

        this.processamentoExtra(evento, registro);
        console.log(`[Framework] notificaĂ§ĂŁo enviada para ${destinatarios.length} destinatĂĄrio(s)`);
        return registro;
    }

    protected validarDestinatarios(destinatarios: Usuario[]): void {
        if (destinatarios.length === 0) {
            throw new Error('A notificaĂ§ĂŁo precisa ter ao menos um destinatĂĄrio.');
        }
    }

    protected despachar(
        destinatarios: Usuario[],
        conteudo: ConteudoNotificacao,
        prioridade: PrioridadeNotificacao
    ): RegistroNotificacao {
        console.log(
            `[Framework] despachando via ${conteudo.canal} com prioridade ${prioridade}: ${conteudo.assunto}`
        );

        return {
            destinatarios,
            conteudo,
            prioridade,
            enviadaEm: new Date(),
        };
    }

    protected abstract definirDestinatarios(evento: EventoNotificacao): Usuario[];
    protected abstract montarConteudo(evento: EventoNotificacao): ConteudoNotificacao;
    protected abstract definirPrioridade(evento: EventoNotificacao): PrioridadeNotificacao;
    protected processamentoExtra(evento: EventoNotificacao, registro: RegistroNotificacao): void { }
}
```

O cliente nĂŁo chama esses passos individualmente; ele apenas fornece as implementaĂ§Ă”es dos hot spots. Isso evidencia a **InversĂŁo de Controle** e o **Hollywood Principle**.

**Hot spots obrigatĂłrios:** `definirDestinatarios()`, `montarConteudo()` e `definirPrioridade()` variam conforme o tipo de evento do OrganizeSeuGrupo.

**Hot spot opcional:** `processamentoExtra()` permite registrar histĂłrico, auditoria ou telemetria sem obrigar todas as notificaĂ§Ă”es a implementarem esse comportamento.

## 8. Senso CrĂ­tico

### BenefĂ­cios observados

- **ReutilizaĂ§ĂŁo efetiva:** o nĂșcleo do framework Ă© compartilhado sem duplicaĂ§ĂŁo entre os diferentes tipos de upload, as diferentes estratĂ©gias de busca e os diferentes eventos de notificaĂ§ĂŁo do OrganizeSeuGrupo.

- **Acoplamento ao contrato:** mudanĂ§as nas interfaces exportadas (ex.: adicionar parĂąmetro a `validarConteudo`) quebram todos os clientes, pois exige versionamento cuidadoso.
- **Custo de extensĂŁo:** cada novo tipo de notificaĂ§ĂŁo exige uma subclasse concreta. Isso Ă© adequado para demonstrar Framework Caixa Branca, mas pode gerar muitas classes se o domĂ­nio crescer sem critĂ©rios.

### 9.3. Frozen e Hot Spot Internos ao PrĂłprio MĂłdulo

A reutilizaĂ§ĂŁo nĂŁo aparece sĂł nos frameworks de terceiros â a arquitetura em camadas adotada no mĂłdulo cria **pontos congelados e quentes dentro do prĂłprio cĂłdigo**, espelhando a mesma distinĂ§ĂŁo da SeĂ§ĂŁo 3.2:
